## What changed this session (commits a532e05..a585f7e on main)

- **iOS AutoFill bug on sign-up form**: iOS's "Automatic Strong Password"
  cover view was getting stuck/uneditable on random fields (not just
  password ones). Fixed with `textContentType="oneTimeCode"` on every field
  in the sign-up form.
- **Password policy rewritten to NIST SP 800-63B** (`src/lib/password.ts`):
  8–64 chars, no forced upper/lower/number/special composition, blocklist
  of common passwords, rejects repeated/sequential runs (`aaaa`, `1234`),
  rejects passwords containing the username/email. Live green/red checklist
  UI extracted to `src/components/password-checklist.tsx`, used by both
  sign-up and password reset.
- **Email confirmation was fundamentally broken, root-caused, and rebuilt**:
  1. First bug: Supabase's Site URL was the default `localhost:3000`
     (nothing listens there) — fixed by deep-linking, but that turned out
     insufficient.
  2. Real root cause: **iOS Mail's Link Tracking Protection silently
     prefetches links in emails**, consuming Supabase's one-time
     confirmation token before the user ever taps it. Confirmed via
     Supabase auth logs (a link got consumed ~47s after send, long before
     any human tap).
  3. Fix: abandoned link-based confirmation entirely, switched to a
     6-digit-code flow (`supabase.auth.verifyOtp`). New screen
     `src/app/(auth)/verify-email.tsx`. Requires the Supabase email
     template to show `{{ .Token }}` as visible text (not inside a link
     href — first attempt at editing the template put it in an `href`,
     which doesn't render as visible text — fixed).
  4. Note: the OTP code length is **not fixed at 6 digits** — GoTrue
     doesn't zero-pad it, so it's been seen at 7 and 8 digits in testing.
     The verify screen does not cap input length; don't reintroduce a
     `maxLength` assumption here.
- **Password reset converted to the same OTP flow** (`reset-password.tsx`).
  Non-obvious wrinkle: `verifyOtp({ type: "recovery" })` establishes a real
  session immediately, which would make the root router jump straight to
  the tabs mid-flow before the user sets a new password. Fixed with a new
  `needsPasswordReset` flag on `AuthProvider` (set on the `PASSWORD_RECOVERY`
  auth event, cleared after `updateUser` succeeds) — same pattern as the
  existing `needsProfileCompletion` guard for Apple first-sign-in.
- Removed now-dead code: the `exchangeCodeForSession` deep-link listener and
  `emailRedirectTo`/`redirectTo` params — no longer relevant now that both
  auth emails use codes, not links.
- Shared helpers extracted for reuse across the two OTP screens:
  `src/lib/resend-cooldown.ts`, `src/hooks/use-countdown.ts`.
